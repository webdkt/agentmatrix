# Session Manager

Session Manager 负责管理邮件的会话视图。它将零散的邮件组织成有结构的会话，支持查询、分页和归档。

---

## 双视角设计

AgentMatrix 中的 Session 采用双视角设计。一封邮件同时携带两个会话 ID：

- **发件人会话 ID**：发件人视角下的会话标识
- **收件人会话 ID**：收件人视角下的会话标识

这意味着同一个对话在发送方和接收方各有一个 Session。两个 Session 指向同一组邮件，但组织方式可能不同。

**为什么这样设计？**

因为 Agent（和人）需要按自己的视角组织邮件。用户可能希望按主题分组，而 Agent 可能希望按任务分组。双视角让每个参与者自主决定如何组织自己的会话。

---

## Session 的构成

一个 Session 由以下信息组成：

- **会话 ID**：唯一标识
- **主题**：通常取自会话中第一封邮件的主题
- **最后邮件时间**：用于排序和显示
- **参与者**：参与该会话的 Agent 名称列表
- **邮件列表**：属于该会话的所有邮件

Session 数据不是独立存储的，而是从邮件数据库中动态构建的。查询某个 Agent 的会话列表时，系统从数据库中筛选该 Agent 作为发件人或收件人的所有邮件，按会话 ID 分组，然后聚合出会话元信息。

---

## 与 Task 的关系

Session 和 Task 是两个不同层面的概念：

- **Session** 是单 Agent 的视角，用于组织邮件展示
- **Task** 是跨 Agent 的客观工作单元，用于追踪任务流程和文件隔离

一个 Task 可能包含多个 Session（涉及多个 Agent），一个 Session 只属于一个 Task（通过邮件的 task_id 字段关联）。

当用户查看与某个 Agent 的对话历史时，看到的是 Session。当系统需要追踪一个任务在多个 Agent 之间的流转时，使用的是 Task。

---

## 查询与分页

Session Manager 支持以下查询操作：

- **列出某 Agent 的所有会话**：用于在界面显示会话列表
- **获取某会话的邮件历史**：按时间顺序返回邮件，支持分页
- **按任务 ID 查询相关会话**：追踪某个任务涉及的所有对话
- **搜索邮件内容**：全文搜索邮件正文和主题

分页采用基于游标的方式，而不是简单的页码。这是因为邮件数据是动态增长的，传统页码在数据变化时会导致重复或遗漏。

---

## 附件的文件系统隔离

附件存储与 Session/Task 紧密相关。文件系统结构如下：

```
MatrixWorld/
└── workspace/
    └── agent_files/
        └── {agent_name}/
            └── work_files/
                └── {task_id}/
                    └── attachments/
                        └── {filename}
```

- 按收件 Agent 分目录：不同 Agent 的文件互不干扰
- 按任务 ID 分目录：不同任务的文件互不干扰
- 附件命名保留原始文件名，冲突时自动重命名

当 Agent 在容器模式下运行时，这个目录会被挂载到容器内部，Agent 在容器内的文件操作实际发生在这个隔离的目录中。

---

## 会话归档

长期运行的系统会积累大量会话数据。Session Manager 支持会话归档：

- 将旧的会话标记为归档状态
- 归档的会话不再出现在默认列表中，但仍可搜索和查看
- 归档会话的附件可以迁移到冷存储以节省空间

归档策略可以按时间（如 30 天前的会话自动归档）或按手动触发配置。

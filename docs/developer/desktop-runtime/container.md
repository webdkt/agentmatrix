# 容器隔离

AgentMatrix 支持让每个 Agent 运行在独立的容器中，实现文件系统和进程级别的隔离。

---

## 设计目的

在多 Agent 协作的场景中，不同 Agent 可能执行不可预测的操作：读写文件、执行命令、安装依赖。如果没有隔离：

- Agent A 可能意外删除 Agent B 的文件
- 一个 Agent 执行的恶意或错误命令可能影响整个系统
- 不同 Agent 的依赖环境可能冲突

容器隔离为每个 Agent 提供一个独立的运行环境，解决这些问题。

---

## 容器运行时

系统支持两种容器运行时：

### Podman（推荐）

Podman 是默认选项。相比 Docker，Podman：
- 不需要守护进程，更轻量
- 在 Windows 上支持更好
- 安全性更高（无需 root 权限）

### Docker

如果系统检测到 Docker 但没有 Podman，会自动回退到 Docker。Docker 的功能与 Podman 等效，只是运行时需要 Docker 守护进程。

### 自动检测

Runtime 启动时会自动检测可用的容器运行时：
1. 检查用户是否通过环境变量指定了运行时类型
2. 检查 `CONTAINER_RUNTIME` 环境变量
3. 自动检测（Podman 优先，Docker 次之）
4. 如果都没有，回退到本地模式

## 容器的生命周期

### 启动

当 Agent 需要执行第一个动作时，容器被创建并启动：
1. 创建容器实例
2. 挂载 Agent 的工作目录到容器内
3. 启动容器内的 Python 运行时
4. Agent 的 Action 在容器内执行

### 运行

容器运行期间：
- 文件操作在容器内的挂载目录中进行
- Shell 命令在容器内执行
- 网络访问通过容器的网络栈

### 休眠

当 Agent 空闲一段时间（没有新邮件需要处理）时，容器可以被休眠：
- 停止容器进程
- 保留容器实例和文件系统状态
- 释放 CPU 和内存资源

### 唤醒

当 Agent 收到新邮件时，容器被自动唤醒：
- 启动已停止的容器
- 恢复之前的文件系统状态
- Agent 继续处理新邮件

---

## 本地模式

如果没有容器运行时，或用户选择不使用容器，Agent 会在本地模式运行：

- 文件操作在宿主机上直接进行
- 命令在宿主机上执行
- 工作目录仍然按 Agent 和任务隔离

本地模式适用于：
- 开发调试阶段
- 没有容器环境的系统
- 对隔离要求不高的场景

本地模式和容器模式可以在运行时切换，不需要重新安装。

---

## 工作目录挂载

每个 Agent 在容器中有独立的工作目录。挂载规则：

- Agent 的工作目录（`workspace/agent_files/{agent_name}/`）挂载到容器内的固定路径
- 容器内的文件操作实际发生在宿主机的工作目录中
- 容器停止后，文件仍然保留在宿主机上

这种设计使得：
- Agent 在容器重启后仍然能看到之前的文件
- 用户可以在宿主机上直接查看和修改 Agent 的文件
- 多个容器实例可以共享同一个工作目录（但不推荐同时运行）

---

## 网络访问

容器内的 Agent 可以访问外部网络（用于网页搜索、API 调用等），但默认不能访问宿主机的内部服务。如果需要访问宿主机的本地服务（如本地部署的 Ollama），需要额外配置网络映射。

---

## 安全性

容器隔离提供了基本的安全边界，但不是绝对安全的：

- 容器逃逸（breakout）在理论上是可能的
- 恶意的 Skill 代码仍可能在容器内造成损害
- 挂载的工作目录在宿主机上是可访问的

对于运行不可信代码的场景，建议：
- 使用受限的用户权限运行容器
- 定期清理和重建容器镜像
- 监控容器的资源使用和异常行为

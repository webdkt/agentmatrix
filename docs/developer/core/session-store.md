# SessionStore — 会话持久化

SessionStore 负责保存和恢复 MicroAgent 的对话历史。它是 AgentMatrix 实现「重启后自动恢复」和「长任务不丢失」的基础。

---

## 职责

SessionStore 只做三件事：

1. **保存**：将当前对话历史（System Message、User Messages、Assistant Messages）写入持久化存储
2. **加载**：根据会话 ID 读取之前保存的对话历史
3. **清理**：在会话结束时删除不再需要的数据（可选）

SessionStore 不干预对话内容，它只是对话历史的「仓库」。

---

## 接口设计

SessionStore 是一个抽象接口，不同场景可以有不同的实现：

- **文件实现**：将对话历史保存为 JSON 文件。简单、可移植、易于调试。
- **数据库实现**：将对话历史保存到 SQLite 或 PostgreSQL。适合多实例部署和大量数据。
- **内存实现**：只在内存中保存，进程结束数据丢失。适合临时任务和测试。

Shell 实现者选择自己场景合适的持久化方式，Core 层只通过统一接口使用。

---

## 持久化时机

对话历史不是每轮都保存，而是在以下时机触发：

- **Checkpoint 时**：MicroAgent 执行到检查点时，Shell 可以选择保存当前状态
- **任务完成时**：MicroAgent 完成执行后，完整对话历史被保存
- **系统关闭时**：Runtime 在关闭前批量保存所有活跃会话
- **压缩后**：上下文压缩完成后，新的消息结构被保存

这种策略平衡了数据安全性和 I/O 开销。

---

## 恢复流程

系统重启后的恢复流程：

1. Runtime 启动，加载所有 Agent 配置
2. 每个 Agent 初始化时，从 SessionStore 加载自己未完成的会话
3. 如果有未完成的会话，Agent 恢复到暂停状态，等待用户决定是继续还是放弃
4. 用户发送新邮件时，Agent 创建新的 MicroAgent，从 SessionStore 加载历史对话作为上下文

这意味着即使系统在 Agent 执行任务到一半时崩溃，重启后也能恢复到最近的安全检查点，不会丢失大量进度。

---

## 与压缩的协作

SessionStore 保存的是压缩后的对话历史。当上下文触发自动压缩后，MicroAgent 的消息列表会被重构（System Message + 原始用户消息 + Working Notes），然后保存到 SessionStore。

这样即使一个任务运行了很长时间、经历了多次压缩，SessionStore 中保存的始终是最紧凑、最可恢复的状态。

---

## 实现建议

如果你正在实现自己的 Shell，关于 SessionStore 的建议：

- **至少实现文件持久化**：内存实现虽然简单，但数据会在进程结束时丢失，不适合生产环境
- **考虑并发**：如果多个 MicroAgent 可能同时读写同一个会话（虽然罕见），需要处理并发冲突
- **定期清理**：长期运行的系统会产生大量历史会话，考虑设置保留策略自动清理旧数据
- **备份敏感数据**：如果对话中包含敏感信息，考虑加密存储

# 自动压缩机制

AgentMatrix 允许 Agent 执行非常长的任务（几小时甚至几天），而不会让上下文窗口溢出。这是通过自动压缩机制实现的。

---

## 问题

LLM 有上下文长度限制。当 Agent 执行复杂的多步骤任务时，对话历史会不断增长：

- 用户的初始请求
- Brain 的多次思考
- 多次 Action 执行的结果
- 中间的澄清和修正
- 子任务的返回结果

当这些内容累计超过模型的上下文上限时，后续请求会失败或被迫截断，导致 Agent 丢失早期的重要上下文。

---

## 解决方案：Working Notes

AgentMatrix 的解决方案不是在消息数量上硬截断，而是让 LLM 自己生成一个压缩后的状态快照，称为 Working Notes。

### 触发条件

当 MicroAgent 的对话历史估算 token 数超过预设阈值（默认约 20 万 tokens，可配置）时，压缩机制自动触发。

### 压缩过程

1. MicroAgent 暂停执行循环
2. 将当前完整的对话历史发送给 Brain
3. 要求 Brain 生成一段 Working Notes，总结到目前为止的所有关键信息
4. Brain 生成的 Working Notes 是一段结构化的 Markdown 文本
5. MicroAgent 用这段 Working Notes 替换掉大部分历史消息

### 重建后的消息结构

压缩完成后，MicroAgent 的对话历史变成：

- System Message（保留不变）
- 最初的 User Message（保留，作为任务起点）
- Working Notes（新生成的状态快照）
- 最近的几条消息（保留，保持短期上下文连贯性）

这样既保留了任务的完整背景，又把消息总量控制在安全范围内。

---

## Working Notes 的格式

Working Notes 的格式不是固定的。系统只提供大致的指导（"总结关键信息、保留重要结论、丢弃中间过程"），具体的结构由 LLM 根据对话类型自主决定。

例如：

- 对于研究类任务，Working Notes 可能是分主题整理的要点列表
- 对于编程任务，可能是已完成的代码模块和待办事项
- 对于创意写作任务，可能是情节大纲和人物设定

这种灵活性比固定模板更有效，因为 LLM 知道哪些信息对当前任务最重要。

---

## 与 SessionStore 的协作

压缩完成后，新的消息结构会被保存到 SessionStore。这意味着：

- 如果系统在压缩后立即重启，恢复时加载的是已压缩的状态
- 旧的完整历史可以选择保留（用于调试）或丢弃（节省空间）
- 多次压缩会产生多个版本的 Working Notes，系统保留最新的

---

## 可配置性

压缩行为可以通过以下方式调整：

- **触发阈值**：修改 token 上限，控制压缩的敏感度
- **保留消息数**：控制压缩后保留的最近消息条数
- **Working Notes 生成策略**：Shell 实现可以自定义 Prompt，影响 Working Notes 的风格和详细程度
- **完全关闭**：在特定场景下可以禁用自动压缩（不推荐用于长任务）

---

## 局限性

压缩机制虽然有效，但不是万能的：

- 如果任务需要精确引用早期对话中的具体数字或文本，压缩后可能丢失这些细节
- 极度复杂的任务可能需要多次压缩，每次压缩都是一次信息损失
- Working Notes 的质量取决于 Brain 模型的能力，弱模型可能生成不完整的总结

对于需要精确追溯每一步的任务，建议在关键节点让 Agent 把重要信息写入文件或长期记忆，而不是完全依赖上下文。

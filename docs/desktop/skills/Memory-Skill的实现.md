# Memory Skill 的实现

## Memory Skill 的功能

Memory Skill 提供基于邮件历史的记忆搜索能力。当智能体需要回忆之前讨论过的事情、决策、项目细节等信息时，可以调用 Memory Skill 来搜索历史对话。

Memory Skill 是嵌套调用和并行执行的典型应用，展示了如何把一个复杂任务分解成多个子任务并行处理。

## 搜索流程概览

Memory Skill 的搜索采用了"关键词提取 + 并行搜索 + 嵌套阅读"的三层架构：

```
用户提问："上次讨论的预算是多少？"
    │
    ├─> 第一步：关键词提取
    │     └─> LLM 分析问题，提取关键词分组
    │         结果：[["预算", "讨论"], ["预算"], ["资金"]]
    │
    ├─> 第二步：并行搜索每组关键词
    │     ├─> 关键词组1: ["预算", "讨论"] ─┐
    │     ├─> 关键词组2: ["预算"] ───┤─> 三组并行
    │     └─> 关键词组3: ["资金"] ─┘
    │
    └─> 第三步：嵌套阅读会话
       （对每个关键词组）
          ├─> 子智能体1 ─┬─> session A (阅读邮件)
          │           └─> session B (阅读邮件)
          │           └─> session C (阅读邮件)
          │
          ├─> 子智能体2 ─┬─> session D (阅读邮件)
          │           └─> session E (阅读邮件)
          │
          └─> 子智能体3 ─┬─> session F (阅读邮件)
                      └─> session G (阅读邮件)

       第一个找到答案 → 立即返回，取消所有其他搜索
```

## 关键词提取机制

搜索的第一步是从用户问题中提取关键词分组。Memory Skill 使用大语言模型来分析问题，提取最多 3 组关键词，按优先级排序。

关键词分组遵循 OR-of-AND 的逻辑：
- **组内是 AND 关系**：一组中的所有关键词必须同时匹配
- **组间是 OR 关系**：按顺序尝试，找到就停止

例如问题"A项目的预算讨论"：
- 第1组（最精确）：`["A项目", "预算"]`
- 第2组（宽泛）：`["A项目"]`
- 第3组（同义词）：`["A", "预算方案"]`

这种分层搜索策略既保证了精度（先用最精确的组合），又提供了回退方案（如果精确组合没有结果）。

## 并行搜索的关键词组

每组关键词由一个独立的任务并行搜索。

```python
# 创建 3 个并行任务
tasks = [
    asyncio.create_task(self._search_one_group(question, 0, ["预算", "讨论"])),
    asyncio.create_task(self._search_one_group(question, 1, ["预算"])),
    asyncio.create_task(self._search_one_group(question, 2, ["资金"])),
]

# 等待第一个完成的任务
while pending:
    done, pending = await asyncio.wait(pending, return_when=asyncio.FIRST_COMPLETED)
    for task in done:
        result = await task
        if result:  # 找到答案
            for p in pending:  # 取消所有其他任务
                p.cancel()
            return result  # 立即返回
```

这种"竞速"模式确保了搜索效率：第一个找到答案的立即返回，其他搜索被取消，节省时间和资源。

## 嵌套阅读：子智能体的创建

每个关键词组会搜索多个会话。对于每个会话，系统创建一个专门的子智能体来阅读邮件。

```
关键词组: ["预算", "讨论"]
    │
    ├─> 子智能体 A1 (session_id_1, batch_1)
    │     └─> 读取邮件 → 思考 → 调用动作
    │
    ├─> 子智能体 A2 (session_id_1, batch_2)
    │     └─> 读取邮件 → 思考 → 调用动作
    │
    ├─> 子智能体 A3 (session_id_2, batch_1)
    │     └─> 读取邮件 → 思考 → 调用动作
    │
    └─> ... 更多子智能体
```

### 子智能体的技能组合

子智能体只使用两个技能：`["memory.memory_reader", "file"]`。

- **memory.memory_reader**：提供 4 个专门的动作（见下文）
- **file**：提供文件操作能力（虽然在这个场景中可能用不到）

这种精简的技能组合减少了 System Prompt 的长度，让子智能体专注于阅读任务。

### 子智能体的任务描述

每个子智能体收到一个结构化的任务描述：

```
你的任务是阅读以下邮件，找到要回答问题的答案。

[待回答问题]
上次讨论的预算是多少？

[搜索关键词]
预算, 讨论

[已记录的笔记]
无

[邮件内容（第 1/3 批）]
[2024-01-15 10:30] user1: 请问一下预算情况
[2024-01-15 10:32] agent1: 预算大约是50万
...
```

### 子智能体的退出动作

子智能体有 4 个退出动作，决定何时停止阅读：

1. **provide_final_answer**：已找到完整答案，提供答案并退出
2. **read_next_batch**：当前批没有完整答案，继续读下一批
3. **session_irrelevant**：当前会话与问题无关，跳过这个会话

这种设计让子智能体可以自主决策何时停止，而不是阅读完所有邮件。

### 邮件分批处理

如果会话包含很多邮件，系统会分批处理，每批 20 封邮件。

```
session_1 (共 65 封邮件)
    │
    ├─> 批次 1 (邮件 1-20)
    │     └─> 子智能体阅读 → 决定 read_next_batch
    │
    ├─> 批次 2 (邮件 21-40)
    │     └─> 子智能体阅读 → 决定 read_next_batch
    │
    ├─> 批次 3 (邮件 41-60)
    │     └─> 子智能体阅读 → 决定 read_next_batch
    │
    └─> 批次 4 (邮件 61-65)
          └─> 子智能体阅读 → 决定 provide_final_answer 或 session_irrelevant
```

这种分批处理避免了一次性加载太多邮件到上下文中，保持了可控的 token 消耗。

## Memory Reader 技能的动作

Memory Reader 是专门为子智能体设计的技能，提供 4 个动作：

### take_note

记录有价值的发现到笔记中。可以多次调用，所有笔记被累积。

### provide_final_answer

提供最终答案并退出阅读。这是结束动作，调用后阅读会结束。

### read_next_batch

继续阅读下一批邮件。这是一个退出动作，调用后当前批次结束，进入下一批。

### session_irrelevant

跳过当前会话。这是一个退出动作，表示当前会话与问题无关。

这 4 个动作让子智能体可以灵活地控制阅读流程，而不是机械地读完全部邮件。

## 会话选择策略

对于每个关键词组，系统搜索数据库，找到匹配的会话。然后选择 hits 最多的前 5 个会话进行阅读。

```
数据库搜索结果（关键词："预算", "讨论"）
    │
    ├─> session_1: 15 hits (邮件中关键词出现次数)
    ├─> session_2: 12 hits
    ├─> session_3:  8 hits
    ├─> session_4:  5 hits
    ├─> session_5:  3 hits
    └─> ... 其他会话
         │
         └─> 只读前 5 个，按优先级排序
```

这种策略确保了最相关的会话优先被阅读，同时限制了阅读范围，避免浪费时间在不相关的会话上。

## 笔记累积机制

子智能体在阅读过程中使用 `take_note` 动作记录有价值的发现。这些笔记在批次之间累积。

```
批次 1:
    ├─> 阅读 → 发现"预算50万" → take_note("预算50万")
    └─> read_next_batch

批次 2:
    ├─> 阅读 → 确认信息 → read_next_batch
    └─> (之前的笔记保留)

批次 3:
    ├─> 阅读 → 找到完整答案 → provide_final_answer
    └─> 返回：笔记 + 最终答案
```

累积的笔记帮助子智能体保持上下文，避免在不同批次之间丢失重要信息。

## 完整的并行嵌套执行流程

让我们把所有组件整合起来，看完整的执行流程：

```
Memory Skill (父智能体)
    │
    │ 1. 提取关键词分组
    ├─────────────────────────────> 大语言模型
    │<──────────────────────────────  关键词组: [[g1], [g2], [g3]]
    │
    │ 2. 并行搜索每组
    ├───────────────────> 子任务 G1 (关键词组1)
    │   ├─> 搜索数据库 → 找到5个session
    │   ├─> 创建子智能体G1_S1 → 读session_1 → read_next_batch
    │   ├─> 创建子智能体G1_S2 → 读session_2 → session_irrelevant
    │   └─> ...
    │
    ├───────────────────> 子任务 G2 (关键词组2)
    │   ├─> 搜索数据库 → 找到3个session
    │   └─> 创建子智能体...
    │
    └───────────────────> 子任务 G3 (关键词组3)
        └─> 搜索数据库 → 找到0个session

    │ 3. 等待第一个完成的任务
    │   (假设 G1_S1 找到了答案)
    │
    ├─────────────────────────────> G1_S1: provide_final_answer
    │
    │ 4. 取消其他所有任务
    ├─> cancel(G2)
    ├─> cancel(G3)
    ├─> cancel(G1_S2, G1_S3, ...)
    │
    └─> 5. 返回答案给用户
```

## 性能优化

Memory Skill 在设计时考虑了多个性能优化：

首先是竞速模式。多个关键词组并行搜索，第一个找到答案的立即返回，其他搜索被取消。这大大缩短了平均搜索时间。

其次是会话选择。只选择 hits 最多的前 5 个会话，避免阅读不相关的会话。

然后是分批处理。每批 20 封邮件，避免一次性加载太多邮件。

最后是资源释放。一旦找到答案，所有其他搜索任务被立即取消，释放资源。

## 错误处理和容错

搜索过程有多层容错机制。

如果关键词提取失败（LLM 返回错误格式），系统返回提示信息，让用户重新表述问题。

如果某个关键词组没有找到任何匹配的会话，该组简单返回 None，不影响其他组。

如果子智能体执行出错（如异常），错误被捕获，该会话被跳过，继续搜索其他会话。

如果所有搜索都没有找到答案，系统返回"记忆里没有找到相关信息"，让用户决定下一步。

## 总结

Memory Skill 是嵌套调用和并行执行的完美示例。它展示了如何：

1. **分解任务**：把复杂的搜索任务分解成关键词提取、并行搜索、嵌套阅读
2. **嵌套调用**：为每个会话创建子智能体，专门负责阅读该会话
3. **并行执行**：多个关键词组、多个会话同时搜索
4. **智能决策**：子智能体通过退出动作自主控制阅读流程
5. **资源优化**：竞速模式、会话选择、分批处理

理解 Memory Skill 的实现，有助于理解如何设计复杂的嵌套和并行任务，以及如何通过合理的架构设计提高系统效率。

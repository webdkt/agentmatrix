# 容器与文件系统

## 共享容器架构

AgentMatrix 采用"单容器多用户"架构。所有智能体共享一个 Linux 容器，每个智能体是容器内的一个独立用户。

你可以把这个容器想象成一台"共享电脑"，所有智能体都使用这台电脑。每个智能体有自己的用户账号和主目录，但它们共享同一个操作系统环境。

## 为什么使用共享容器

共享容器有几个重要好处。

首先是资源效率。不需要为每个智能体运行独立的容器，节省了内存和 CPU。所有智能体共享操作系统的开销。

其次是隔离性。每个智能体是容器内的独立用户，有自己的主目录和工作空间。一个智能体无法访问另一个智能体的文件，保证了隔离。

然后是统一的环境。所有智能体使用相同的操作系统和工具，不需要为每个智能体配置环境。简化了部署和维护。

## 容器内的用户管理

当智能体被创建时，系统会在容器内为它创建一个 Linux 用户账号。

用户名从智能体的名字转换而来。由于 Linux 用户名有字符限制（只允许字母、数字、下划线、连字符，最多 32 字符），系统会清理不合规的字符。如果清理后仍不合规，会使用哈希值生成唯一用户名。

重要的是，虽然用户名可能被清理，但所有目录路径都使用智能体的原始名称。这样保证了可读性和可追溯性。

## 智能体的目录结构

每个智能体在容器内有自己的目录空间，位于 `/data/agents/{agent_name}/`。

```
/data/agents/{agent_name}/
├── home/              # 主目录（~）
└── work_files/        # 工作文件目录
    ├── task_id_1/     # 任务1的工作目录
    ├── task_id_2/     # 任务2的工作目录
    └── ...
```

`home` 是智能体的主目录，对应 Linux 用户的 HOME 目录。智能体可以在这里存储配置文件、临时文件等。

`work_files` 存放任务相关的工作文件。每个任务对应一个子目录，以 task_id 命名。

## ~/current_task 软链接

每个智能体在主目录下有一个固定的软链接 `~/current_task`，指向当前任务的工作目录。

```
~/current_task → /data/agents/{agent_name}/work_files/{current_task_id}/
```

这个软链接提供了便捷的访问方式。智能体不需要记住完整路径，只需要使用 `~/current_task` 就能访问当前任务的工作目录。

当切换任务时，系统会重新创建这个软链接，让它指向新的任务目录。

## 宿主机与容器的映射

容器内的 `/data/agents/` 目录从宿主机映射而来。

```
宿主机: workspace_root/agent_files/{agent_name}/
容器内: /data/agents/{agent_name}/
```

这个映射通过 volume mount 实现。当智能体在容器内创建文件时，文件实际被写入宿主机的 `workspace_root/agent_files/` 目录。

这个映射有几个作用：
1. 文件持久化：容器重启后文件不会丢失
2. 外部访问：宿主机上的其他进程可以访问这些文件
3. 备份和恢复：可以直接备份宿主机的目录

## 工作区切换

智能体可以同时处理多个任务，每个任务有独立的工作目录。

当收到新任务或切换任务时，智能体会调用 `switch_workspace` 方法：
1. 在宿主机创建新的任务目录（如果不存在）
2. 在容器内重新创建 `~/current_task` 软链接，指向新目录

这样，智能体在任何时刻都通过 `~/current_task` 访问当前任务的工作目录，不需要记住完整路径。

## File Skill 的工作方式

File Skill 提供文件操作能力，所有操作都在容器内执行。

当智能体调用文件操作动作时（如 `read_txt_file`），File Skill 会：
1. 接收路径参数（如 `~/current_task/data.txt`）
2. 在容器内执行命令（如 `cat ~/current_task/data.txt`）
3. 返回输出给智能体

智能体看到的路径是容器内的路径。它不知道也不需要知道宿主机的实际路径。

File Skill 支持的操作包括：
- 列出目录内容
- 读取文件
- 写入文件
- 搜索文件
- 执行 shell 命令（通过 bash 动作）

## 路径约定

为了保持一致性，系统有一些路径约定。

**当前任务的工作目录**：`~/current_task`
- 指向当前 session 的任务目录
- 任务的主要输出应该放在这里

**临时文件目录**：`~/current_task/tmp`
- 存放任务执行过程中的临时文件

**输出目录**：`~/current_task/output`
- 存放任务的最终输出

**附件目录**：`~/current_task/attachments`
- 存放发送给他人的附件（需要保留副本）

这些约定让不同智能体的工作方式保持一致，便于理解和协作。

## 智能体之间的隔离

虽然共享容器，但智能体之间是隔离的。

首先是用户隔离。每个智能体是独立的 Linux 用户，只能访问自己的主目录和工作目录。一个智能体无法读取或修改另一个智能体的文件。

然后是进程隔离。每个智能体以自己的用户身份运行命令，只能管理自己的进程。

最后是资源隔离。容器级别的资源限制（CPU、内存）适用于所有智能体，防止某个智能体占用过多资源。

## 安全考虑

共享容器架构有一些安全考虑。

所有智能体共享同一个内核，容器级别的安全漏洞可能影响所有智能体。因此，容器应该保持最小化，只安装必要的工具。

智能体之间的隔离依赖于 Linux 用户权限。如果某个智能体的代码被攻破，攻击者可能尝试提权访问其他智能体的文件。因此，应该遵循最小权限原则。

宿主机上的 `workspace_root/agent_files/` 目录包含所有智能体的文件，应该设置合适的权限，防止未经授权的访问。

## 容器的生命周期

容器的生命周期由系统管理。

容器在系统启动时被创建或连接。如果容器不存在或已停止，系统会启动容器。

容器在所有智能体被销毁后仍然保留。这样可以避免频繁创建销毁容器的开销。下次启动系统时，容器可以直接复用。

容器可以被手动停止和重启，但通常不需要。系统会自动管理容器的状态。

## 总结

容器与文件系统通过共享容器、独立用户、目录映射等机制，实现了高效的资源利用和良好的隔离性。

理解这些机制，有助于理解智能体如何操作文件，如何保持隔离，以及系统如何管理多任务场景。

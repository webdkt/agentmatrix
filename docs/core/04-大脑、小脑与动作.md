# 大脑、小脑与动作

## 双脑架构的设计理念

AgentMatrix 中的智能体采用了一种独特的"双脑"架构：大脑和小脑。这个设计借鉴了生物学的概念，但在这里有特定的含义。

大脑负责复杂的认知任务：理解邮件内容、推理要采取的行动、规划执行步骤、生成回复文本。大脑是一个大语言模型客户端，通过调用大语言模型的 API 来实现智能。

小脑负责具体的执行协调：解析动作的参数、验证参数的有效性、与大语言模型协商缺失的参数、确保动作能够被正确执行。小脑也是大语言模型客户端，但它通常配置为使用一个能力较弱、成本较低的模型。

这种分工很有道理。大语言模型擅长理解自然语言、进行常识推理、生成连贯的文本，但调用强大的模型成本较高。小脑处理的工作（参数解析和协商）比较机械性，不需要很强的推理能力，使用较便宜的模型就足够了。这样可以在保证功能的同时降低成本，让昂贵的大模型资源专注于它最擅长的复杂推理任务。

## 大脑：LLMClient 的职责

大脑由 LLMClient 类实现，它是智能体的思考引擎。当智能体收到邮件需要处理时，它会调用大脑的方法来获取"想法"。

大脑的核心方法是"思考"，它接收一系列消息（对话历史），发送给大语言模型，然后返回模型的回复。这看起来很简单，但实际应用中有很多细节需要处理。

首先是模型选择。AgentMatrix 支持多种大语言模型服务，比如 OpenAI 的 GPT 系列、Anthropic 的 Claude 系列、Google 的 Gemini 系列。LLMClient 可以根据配置选择使用哪个模型。不同的模型有不同的特点，有些擅长推理，有些速度快，有些成本低，系统可以根据任务需求灵活选择。

其次是流式和非流式调用。非流式调用是一次性获取完整回复，适合处理简单任务。流式调用是逐步获取回复内容，适合需要长时间生成的任务，可以实时显示进度，提供更好的用户体验。

然后是错误处理。大语言模型的 API 调用可能会失败，网络可能会中断，模型可能会返回错误。大脑有重试机制，遇到暂时性错误会自动重试，最多尝试几次后才放弃。

最后是智能重试模式（think_with_retry），这是大脑最强大的功能之一。

## think_with_retry：智能重试模式

在调用大语言模型时，一个常见的问题是模型返回的格式不符合预期。比如你要求模型返回 JSON 格式的动作列表，但模型返回了带有解释性文字的文本，导致解析失败。

传统的做法是失败后就直接报错，或者简单重试几次同样的问题。但 think_with_retry 采用了更智能的方式：它会给模型反馈，告诉它哪里出错了，让它重新生成。

具体来说，think_with_retry 接收一个解析器函数，这个函数负责把模型的原始回复解析成结构化的数据。如果解析成功，就返回解析结果。如果解析失败，解析器会生成一条反馈信息，说明具体哪里出错了、期望的格式是什么。

然后 think_with_retry 会把原始回复和反馈信息加入消息历史，再次调用模型。这样模型可以看到自己之前的回答和错误反馈，理解哪里不对，然后生成更符合要求的回复。

这个过程可以重复多次，最多达到设定的重试上限。每次重试都会积累更多的上下文，模型会逐渐理解期望的格式。这种模式大大提高了大语言模型输出结构化数据的可靠性，不需要精心设计提示词就能获得稳定的输出。

## 小脑：Cerebellum 的职责

小脑由 Cerebellum 类实现，它是智能体的动作协调器。当大脑决定要执行某个动作时，小脑负责确保这个动作能够被正确执行。

小脑的核心功能是参数解析和协商。每个动作都有一个参数模式，定义了这个动作需要哪些参数、每个参数的类型、是否必需、默认值是什么。当大脑决定执行某个动作时，它会提取动作的描述（可能是自然语言形式），然后交给小脑处理。

小脑首先会尝试从描述中解析出参数值。如果解析成功，它会验证参数值是否符合模式要求：类型是否正确、必需参数是否都提供了、值是否在允许的范围内。如果验证通过，小脑返回解析好的参数，动作就可以被执行了。

但事情往往不会这么顺利。大脑可能没有提供所有必需的参数，或者提供的参数格式不对。这时候小脑需要和大脑进行协商。

## 参数协商机制

参数协商是小脑最独特的功能。当小脑发现参数缺失或无效时，它不会直接报错，而是会生成一个问题，通过自己调用的语言模型来询问这些参数应该是什么值。

小脑使用的模型虽然能力较弱，但对于这种机械性的参数协商任务已经足够。比如，如果一个动作需要文件路径参数，但大脑没有提供，小脑会问"这个动作应该在哪个文件上操作？"模型会根据对话历史推断出合理的文件路径。

小脑收到回答后，再次进行解析和验证。如果这次成功了，动作继续执行。如果还是不行，小脑可以再次提问，最多进行几轮协商。

这种协商机制让系统更加鲁棒。大脑不需要在第一次思考时就规划好所有细节，可以先把大方向确定下来，然后在执行过程中逐步补充细节。这更接近人类的工作方式：我们做决定时往往先确定目标，然后在执行中解决具体问题。

## 动作：智能体的能力单元

动作（Action）是智能体能执行的最小操作单元。每个动作对应一个具体的函数，接收参数，执行操作，返回结果。

动作是智能体能力的体现。一个智能体能做什么，取决于它注册了哪些动作。系统提供了很多内置动作：发送邮件、读写文件、搜索信息、调用工具等。开发者也可以定义自己的动作，扩展智能体的能力。

动作通过装饰器注册。在定义函数时加上动作注册装饰器，指定动作的名字、描述、参数模式，这个函数就成为了一个动作。系统会自动收集所有注册的动作，建立动作注册表，供大脑查询和使用。

动作的描述很重要。大脑是通过阅读动作描述来决定使用哪个动作的。描述应该清晰说明动作的作用、使用场景、参数要求。好的描述可以让大脑更容易做出正确的决策。

## 大脑和小脑的协作

大脑和小脑的协作构成了智能体的完整工作流程。

当智能体收到一封邮件时，它首先构建一个提示，包含系统提示（定义智能体的角色和能力）和对话历史（包含之前会话中的邮件和动作执行结果）。然后把提示发送给大脑，让大脑思考。

大脑收到提示后，通过大语言模型生成一个回复。这个回复可能包含文本内容（对邮件的回复）和动作指令（要执行的动作列表）。大脑使用 think_with_retry 模式，确保能够正确解析出动作列表。

对于每个动作，大脑提取出动作名称和参数描述，然后交给小脑。小脑解析参数，进行验证。如果需要协商，小脑使用自己调用的语言模型来获取缺失的参数信息。

参数准备好后，系统执行对应的动作函数。动作函数执行具体的操作：可能是发送另一封邮件，可能是读写文件，可能是调用外部服务。动作函数返回结果，这个结果会被加入对话历史，供大脑参考。

如果大脑决定执行多个动作，这些动作可以并发执行，以加快速度。所有动作完成后，大脑会再次思考，基于动作执行的结果决定下一步做什么。

这个过程会持续循环，直到大脑认为任务完成，生成最终的回复邮件，结束会话。

## 视觉能力

现代大语言模型不仅支持文本，还支持图像输入。AgentMatrix 的大脑也有视觉能力，可以"看"图片并理解内容。

视觉能力通过单独的方法提供，接收图像数据和提示，返回模型对图像的理解。这可以用于很多场景：分析截图、读取图片中的文字、理解图表内容、描述图像场景等。

视觉能力也支持智能重试模式，和文本思考类似。如果模型的返回格式不符合预期，系统会给出反馈，让模型重新分析。

## 总结

大脑和小脑的双脑架构是 AgentMatrix 智能体的核心。大脑使用强大的语言模型负责理解和推理，小脑使用较便宜的模型负责参数协调和验证。它们通过 think_with_retry 和参数协商机制紧密协作，构成了一个既能深度思考又能精确执行，同时还能控制成本的智能系统。

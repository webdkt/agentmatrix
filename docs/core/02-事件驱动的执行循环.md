# 事件驱动的 MicroAgent 执行循环

## 从收到邮件到执行动作

当一个 BaseAgent 收到一封新邮件时，它不会自己直接处理，而是会创建一个 MicroAgent 来处理这个会话。MicroAgent 会进入一个执行循环，不断地接收事件、思考、执行动作，直到任务完成。

这个执行循环是事件驱动的。什么是事件驱动？意味着 MicroAgent 不是一直运行的，而是在有事件发生时被唤醒，处理完事件后又进入等待状态。事件可以是收到新邮件、某个动作执行完成、某个动作执行失败等。

事件驱动的好处是高效。MicroAgent 不需要占用 CPU 资源来轮询是否有事情要做，而是可以在等待时把资源让给其他任务。当事件发生时，系统会通知 MicroAgent，唤醒它继续工作。

## Signal 事件系统

AgentMatrix 使用 Signal 事件系统来驱动 MicroAgent 的执行循环。Signal 是一个简单的数据结构，包含事件类型和事件数据。

有几种基本的 Signal 类型。邮件信号表示收到了新邮件，这是最常见的信号类型。动作完成信号表示某个动作执行成功，包含了动作的结果数据。动作失败信号表示动作执行出错，包含了错误信息。所有动作完成信号表示一批并发执行的动作都已完成。

MicroAgent 有一个信号队列，所有发给它的事件都会被放入这个队列。执行循环会从队列中取出信号，根据信号类型决定如何处理。

这个设计允许灵活的事件组合。比如，MicroAgent 可以在等待动作执行完成的同时，收到了新邮件。这两个事件都会被放入队列，执行循环会按照队列顺序依次处理。

## 批量信号处理

执行循环的一个关键特性是批量处理信号。当循环被唤醒时，它不会只处理一个信号就返回等待，而是会一次性处理队列中的所有信号。

批量处理有几个好处。首先是效率。如果队列中有多个信号，批量处理可以减少循环次数，提高吞吐量。其次是上下文连贯性。当多个信号被一起处理时，大脑可以看到完整的事件序列，做出更合理的决策。

具体来说，执行循环会先清空信号队列，取出所有待处理的信号。然后把每个信号注入到对话历史中。不同的信号会以不同的方式注入：邮件信号会添加用户的邮件内容，动作完成信号会添加动作的执行结果。

所有信号都注入完成后，执行循环调用大脑进行思考。大脑看到完整的对话历史（包括之前的对话和刚刚注入的所有信号），生成一个回复。这个回复可能包含新的动作指令，也可能只是对事件的确认。

## Think-Negotiate-Act 循环

执行循环的核心是 Think-Negotiate-Act 三个阶段的循环。

Think 阶段，大脑分析当前状态，决定要做什么。它查看对话历史，理解用户的需求，考虑已经执行的动作及其结果，然后规划下一步行动。大脑的输出包含两个部分：文本内容（可能是对用户的回复或中间思考）和动作列表（要执行的动作及参数）。

Negotiate 阶段，小脑对每个动作进行参数解析和协商。如果大脑提供了完整的参数，小脑验证参数后直接准备执行。如果参数缺失或格式不对，小脑会向大脑提问，获取缺失的信息。这个过程可能需要多轮对话，直到所有参数都准备好。

Act 阶段，系统执行所有动作。如果动作之间没有依赖关系，它们可以并发执行，以加快速度。每个动作执行完成后，会生成一个动作完成信号，放入信号队列。动作执行过程中如果出错，会生成动作失败信号。

这三个阶段不断循环：每次循环都处理所有待处理的信号，注入对话历史，触发 Think 阶段，然后 Negotiate 和 Act，生成新的信号。这个循环会持续，直到大脑决定不再执行动作，或者达到某个退出条件。

## 退出条件和动作

MicroAgent 的执行循环不会无限进行下去，它有明确的退出机制。

最简单的退出方式是大脑不生成任何动作指令。如果 Think 阶段没有返回动作，只返回文本内容，系统会认为任务完成，把这段文本作为回复邮件发送给用户，然后结束会话。

更复杂的退出方式是通过退出动作。有些动作被定义为"退出动作"，当这些动作被执行时，执行循环会结束。比如"provide_final_answer"动作表示提供最终答案并结束会话，"session_irrelevant"动作表示当前会话不再相关。

退出动作的设计很重要。在某些场景下，MicroAgent 可能需要执行多个动作，然后在某个特定动作后结束。比如在搜索任务中，MicroAgent 可能先执行几个搜索动作，然后执行"提供最终答案"动作结束。

开发者可以在创建 MicroAgent 时指定退出动作列表。执行循环会检查每个执行的动作是否在退出列表中，如果是，就在动作完成后结束循环。

## 邮件路由到信号队列

你可能有个疑问：当 BaseAgent 收到新邮件时，邮件是如何变成 MicroAgent 的信号的？

这涉及到邮件路由机制。BaseAgent 有一个邮件路由方法，当收到新邮件时，它会判断这封邮件属于哪个会话。如果这是当前活动会话的邮件，它会把邮件转换成一个邮件信号，放入当前 MicroAgent 的信号队列。如果这封邮件属于另一个会话，BaseAgent 会先保存这封邮件，等当前会话结束后再处理。

这种设计确保了会话的隔离性。每个 MicroAgent 只处理自己会话内的邮件，不会被其他会话的邮件打断。当多个会话的邮件同时到达时，BaseAgent 会逐个处理，保证每个会话都能连续完成。

## MicroAgent 与 BaseAgent 的关系

现在我们可以更清楚地理解 MicroAgent 和 BaseAgent 的关系。

BaseAgent 是持久的、有状态的实体。它管理着多个会话，每个会话对应一个 MicroAgent。BaseAgent 负责接收邮件、判断邮件属于哪个会话、创建和管理 MicroAgent、发送回复邮件。

MicroAgent 是临时的、无状态的实体。它被创建来处理一个特定会话，有自己的执行循环，使用从 BaseAgent 继承的大脑和小脑。当会话结束时，MicroAgent 被销毁，它的状态不会保留。

这种分离有很多好处。BaseAgent 可以同时管理多个会话，每个会话由独立的 MicroAgent 处理，互不干扰。MicroAgent 可以嵌套创建，一个 MicroAgent 可以创建另一个 MicroAgent 来处理子任务。BaseAgent 负责持久化和通信，MicroAgent 负责任务执行，职责清晰。

## 错误处理和恢复

执行循环有完善的错误处理机制。

如果大脑调用失败（比如大语言模型 API 出错），系统会记录错误，可以配置是否重试。如果重试多次都失败，循环会结束，错误信息会被发送给用户。

如果某个动作执行失败，会生成动作失败信号。大脑会在下一个 Think 阶段看到这个失败，可以决定如何处理：可能是重试该动作，可能是尝试替代方案，可能是向用户报告错误。

如果整个循环因为未捕获的异常而崩溃，异常会被 BaseAgent 捕获。BaseAgent 会记录错误，可能会尝试重新创建 MicroAgent 来恢复会话，或者向用户报告。

这种多层错误处理确保了单个会话的错误不会影响其他会话，单个 MicroAgent 的错误不会影响整个 BaseAgent。

## 性能考虑

事件驱动的执行循环在性能方面有以下几个实际考虑。

首先是信号队列的设计。信号队列使用异步队列实现，没有设置大小限制，可以无限累积信号。这种设计简化了实现，避免了信号丢失，但也意味着理论上队列可能无限增长。在实际应用中，邮件的到达速率是有限的，通常不会导致问题。但需要注意监控队列深度，如果发现异常增长，可能表明某些动作执行过慢或卡死。

其次是大脑调用的频率和批量处理。每次循环都会调用大语言模型 API，这是主要的成本和延迟来源。系统通过批量处理信号来优化：不是收到一个信号就立即调用大脑，而是等待一小段时间，收集所有待处理的信号，然后一次性调用大脑。这样可以在一次 LLM 调用中处理多个事件，减少调用次数。批量处理的另一个好处是，大脑可以看到完整的上下文（所有相关事件），做出更合理的决策。

然后是动作的执行方式。当大脑生成多个动作时，这些动作被放在一个后台任务中**顺序执行**，而不是并发执行。顺序执行简化了错误处理和结果收集，也避免了并发可能导致的问题（如资源竞争）。虽然并发执行理论上更快，但顺序执行对于大多数场景已经足够高效，因为动作通常是 I/O 密集型的（如调用外部 API），而不是 CPU 密集型的。

接下来是后台任务的调度。动作执行是异步的，在后台任务中运行。当动作完成时，通过信号队列通知主循环。这样主循环不会被长时间阻塞，可以持续响应新事件。后台任务的调度由 Python 的 asyncio 事件循环管理，它会自动在任务之间切换，实现高效的并发。

最后是资源清理机制。当 MicroAgent 结束时，系统会调用技能的清理方法（skill_cleanup），让每个技能有机会释放资源。比如浏览器技能可以关闭打开的标签页，文件技能可以关闭打开的文件句柄。这种机制确保资源被及时释放，防止泄漏。清理是自动的，不需要手动管理。

## 总结

事件驱动的执行循环是 MicroAgent 的核心行为模式。通过 Signal 事件系统、批量处理、Think-Negotiate-Act 循环、退出动作等机制，系统实现了一个高效、灵活、可靠的任务执行模型。

# System Prompt 的构成

## System Prompt 的作用

System Prompt 是发送给大语言模型的第一条消息，它的角色是 "system"。它定义了智能体的角色、能力、行为准则和输出格式。一个好的 System Prompt 能显著影响智能体的表现，让模型更好地理解任务并产生符合预期的输出。

AgentMatrix 中的 System Prompt 是动态生成的，每次调用大语言模型时都会重新构建。这确保了提示始终与当前状态同步，包含了最新的可用动作和上下文信息。

## 两套模式：简化模式和完整模式

System Prompt 有两套模式，分别适用于不同的场景。

简化模式（simple_mode）用于嵌套的 MicroAgent 或子任务。在这种模式下，System Prompt 只包含最核心的信息：智能体的 persona 和可用工具列表。这减少了 token 消耗，让子任务的执行更高效。

完整模式用于顶级 MicroAgent，即直接处理用户任务的智能体。这种模式使用完整的提示模板，包含了更详细的说明、更多的上下文信息和更丰富的指令。

## System Prompt 的模板结构

完整模式的 System Prompt 基于一个 Markdown 模板，模板存储在 `matrix-template/.matrix/configs/prompts/system_prompt.md` 文件中。模板使用 Python 的 string.Template 进行变量替换。

模板分为几个主要部分：

### 1. 身份与环境 (Identity & Environment)

这部分定义了智能体的基本身份和工作环境。包含以下变量：
- `$user_name`：用户的名称
- `$agent_name`：智能体的名称
- `$persona`：智能体的角色描述（从配置中读取）

这部分还包含了一个架构图，说明了智能体、用户和 AgentMatrix 系统之间的关系：智能体思考，AgentMatrix 执行动作，通过邮件与用户沟通。

### 2. 思考方式 (Thinking Approach)

这部分指导智能体如何处理任务和补充信息。它定义了一个优先级顺序：
1. 搜索邮件历史（如果信息涉及过往工作）
2. 对话历史（如果可能在过往讨论过）
3. 外部搜索（如果需要最新信息）
4. 问用户（以上都无法解决时）

这个优先级引导模型先利用已有信息，再寻求外部帮助，最后才打扰用户。

### 3. 记忆使用和维护

这部分介绍了记忆系统的层级结构，从内到外、从快到慢：
- Layer 1：当前记得的（hot cache）
- Layer 2：Scratchpad and Working Notes（快速 RAM）
- Layer 3a：本地文件
- Layer 3b：邮件（持久化）
- Layer 4：互联网（不需要主动管理）

智能体被引导主动管理第 1-3 层级的记忆，把重要信息记入 Scratchpad（第 2 层）。

### 4. 工具使用原则

这部分定义了使用动作（Actions）的基本原则：
- **显式意图**：每个动作都必须有明确的目的
- **参数完备**：从上下文中提取所有必要参数，不要含糊
- **无状态**：动作看不到对话历史或 Working Notes，需要显式传递完整参数
- **单次执行**：每次 `[ACTION]` 中只执行一个动作，复杂任务拆解为多步

### 5. 可用工具箱 (Toolbox)

这是 System Prompt 中最重要的部分，列出了智能体可以使用的所有动作。这部分包含三个子部分：

#### A. 基本技能库 (Basic Skills)

这部分列出了系统内置的动作，按技能（Skill）分组。格式为：

```
**skill_name**: skill 描述
  • action1: action1_short_desc
  • action2: action2_short_desc
  • action3: action3_short_desc
```

智能体被推荐使用 `skill_name.action_name` 格式来调用动作，也可以直接使用 action 名称。

#### B. 扩展技能库 (Procedural Skills)

这部分列出了存放在 `~/SKILLS/` 目录中的扩展技能。这些技能通常是外部脚本或程序，每个技能有自己的目录和 skill.md 描述文件。

智能体被指导如何使用扩展技能：先列目录看有什么技能，再读取 skill.md 判断是否需要，然后按照文档执行（通常通过 bash 命令）。

#### C. 协作网络 (Collaborators)

这部分列出了可以协作的其他智能体（黄色目录）。如果当前智能体无法独立完成任务，可以联系这些智能体寻求帮助。

### 6. 响应协议 (Response Protocol)

这部分定义了智能体的输出格式，包含两个标签：

**[THOUGHTS]** 标签：这是智能体的草稿纸，给自己看的想法和计划。工具看不到这部分内容，格式自由，只需要清晰表达思考过程。

**[ACTION]** 标签：这是给工具执行器看的动作指令。为实现意图而要做的动作，并提供完成该动作需要的全部信息。格式类似伪代码，不需要自然语言描述。

智能体被要求在 THOUGHTS 和 ACTION 之间避免信息重复。例如，如果要写入文件，具体要写的内容必须在 ACTION 中明确提供，而不需要在 THOUGHTS 中念叨一遍。

## action 列表的生成

action 列表（即基本技能库部分）是从智能体的动作注册表（action_registry）中动态生成的。

系统遍历 action_registry 中的 `_by_skill` 字典，这个字典按技能名称组织了所有动作。对于每个技能：

1. 获取技能的描述（_skill_description）
2. 输出技能名称和描述
3. 遍历该技能下的所有动作
4. 对于每个动作，获取其简短描述（_action_short_desc）
5. 格式化为 "• action_name: short_desc" 的形式

这样生成的列表是按技能分组的，每个技能的技能描述和动作描述都是从动作注册表中读取的元数据。

## 扩展技能的发现与注入

扩展技能库部分是通过扫描智能体的 SKILLS 目录动态生成的。

系统会查找智能体目录下的 SKILLS 子目录，遍历其中的所有子目录。每个子目录代表一个技能，目录名即技能名。

对于每个技能目录，系统会查找 skill.md 文件（不区分大小写）。如果找到，会读取文件开头的 frontmatter，提取 description 字段作为技能描述。

然后系统生成一段文本，列出所有可用的扩展技能及其描述。智能体在运行时可以根据这个列表发现和使用扩展技能。

需要注意的是，扩展技能的执行通常需要通过 bash 动作调用外部脚本或命令。

## 协作网络的注入

协作网络（黄色目录）部分是从智能体的 yellow_pages 属性注入的。yellow_pages 包含了可以协作的其他智能体的列表和描述。

这部分帮助智能体了解它可以把任务委派给哪些其他智能体，或者可以向谁寻求帮助。

## 动态生成与无缓存

System Prompt 是在每次调用 `_build_system_prompt()` 方法时动态生成的，没有使用缓存。

这意味着每次智能体思考时，System Prompt 都会重新构建。这确保了提示始终包含最新的信息：
- 如果技能被动态添加或移除，动作列表会立即更新
- 如果黄色目录发生变化，协作网络会立即更新
- 如果 persona 发生变化，身份描述会立即更新

动态生成的代价是每次都需要遍历动作注册表和扫描 SKILLS 目录，有一定开销。但考虑到智能体思考的频率通常不是很高，这个开销是可以接受的。

## System Prompt 的保存

生成的 System Prompt 会被保存在两个地方：
1. MicroAgent 的 system_prompt 属性
2. 如果是顶级 MicroAgent，还会保存在 root_agent.last_system_prompt 属性

这个保存主要用于调试和日志记录，让系统可以查看最近一次使用的 System Prompt 内容。

## 总结

System Prompt 是 AgentMatrix 与大语言模型交互的关键接口。通过模板化设计、动态生成、按技能分组的动作列表、扩展技能发现等机制，系统构建了一个既丰富又灵活的提示体系。

理解 System Prompt 的构成，有助于理解如何设计智能体的 persona，如何组织技能和动作，以及如何引导模型产生符合预期的行为。

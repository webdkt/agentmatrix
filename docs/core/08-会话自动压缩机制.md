# 会话自动压缩机制

## 为什么需要会话压缩

随着对话的进行，会话历史会越来越长。每次智能体思考时，都需要把完整的会话历史作为上下文发送给大语言模型。这带来了几个问题。

首先是成本问题。大语言模型按输入和输出的 token 数量计费，会话历史越长，每次思考的成本就越高。一个长时间运行的会话可能积累了上百轮对话，每次调用模型都要重新发送所有这些内容，费用会快速增长。

其次是性能问题。模型处理长上下文需要更多时间，响应会变慢。用户提问后需要等待更久才能收到回复，体验会下降。

然后是模型能力限制。大语言模型有上下文长度限制。如果会话历史超过了这个限制，模型就无法处理了。即使支持更长上下文的模型，也可能在处理超长历史时出现注意力分散、忘记早期信息等问题。

最后是信息密度问题。长对话中有很多内容是重复的或不重要的。比如问候语、确认消息、中间的错误尝试等。这些信息占用 token，但对当前任务的价值不大。

会话压缩就是为了解决这些问题。通过智能地选择和总结会话内容，在保持关键信息的同时减少 token 消耗，让长对话能够持续进行。

## 压缩触发条件

系统在特定条件下自动触发压缩，不需要手动干预。

第一个触发条件是 token 阈值。系统会估算会话历史的 token 数量，当估算值超过设定的阈值（默认为 32000 tokens）时，系统触发压缩。

这个阈值通常设置在模型上下文限制的 70-80% 之间。这样在接近限制前就开始压缩，留出安全余量。

第二个触发条件是草稿纸积累。草稿纸（scratchpad）是智能体记录要点的地方。当智能体在执行任务时，它可以把重要的发现、决策、约束条件记录在草稿纸上。当草稿纸条目达到一定数量（默认为 8 条）时，系统触发压缩。

为什么用草稿纸作为触发条件？因为草稿纸条目代表了任务进展的关键节点。每次记录草稿纸，意味着智能体有新的发现或做了重要决策。积累了一定数量的关键节点后，会话历史已经比较长了，是时候压缩一下了。

## 压缩的核心机制：Working Notes

压缩不是简单地删除旧消息，而是生成一个结构化的摘要，叫做 Working Notes（工作笔记）。

Working Notes 的生成采用了"液态金属架构"（Liquid Metal）：不固定结构，让大语言模型根据对话场景动态生成适合的结构。

系统会构造一个特殊的提示，包含：
- Scratchpad（草稿纸）：智能体在执行过程中记录的要点
- Session History：当前的对话历史
- Agent Persona：智能体的角色描述（截取前 800 字符）
- 可选的 Focus Hint：指导模型重点关注某方面

提示会引导模型执行以下步骤：
1. 判断有无现存的 Working Notes。如果有，继承其结构并更新内容
2. 场景识别：判断对话的核心模式（任务导向、知识探索、情感咨询、角色扮演等）
3. 基于场景动态定义 Working Notes 的一级标题
4. 状态提取：去噪、消歧、客观记录、保留关键线索

这种动态生成的结构能够适配不同类型的对话场景，而不是使用固定的模板。

## Top-level 和嵌套 MicroAgent 的不同处理

压缩逻辑根据 MicroAgent 的类型有不同的处理方式。

### Top-level MicroAgent（处理用户任务的智能体）

Top-level MicroAgent 的 parent 是 BaseAgent，它直接处理用户的邮件。

在压缩时，Top-level MicroAgent 会：
1. 从 PostOffice 获取 session 的所有邮件
2. 把邮件格式化成紧凑的聊天风格（Email History）
3. 生成 Working Notes
4. 构建新的 user message：Email History + Working Notes
5. 保留原有的 system message（如果有）
6. 最终的 messages 结构：[system?, user(Email History + Working Notes)]

这种设计的理念是：邮件记录始终可读，Working Notes 聚焦于执行层、发现层、调整层的信息，不需要重复沟通层的内容。

### 嵌套 MicroAgent（子任务智能体）

嵌套 MicroAgent 的 parent 是另一个 MicroAgent，它处理的是子任务。

在压缩时，嵌套 MicroAgent 会：
1. 找到第一个 user message（原始的子任务请求）
2. 生成 Working Notes
3. 构建新的 user message：原始内容 + Working Notes
4. 如果之前已有 Working Notes，替换掉旧的，保留原始内容
5. 保留原有的 system message（如果有）
6. 最终的 messages 结构：[system?, user(原始内容 + Working Notes)]

这种设计确保了子任务的上下文不丢失，同时通过 Working Notes 保留了执行过程中的关键信息。

## Scratchpad（草稿纸）的作用

Scratchpad 是压缩机制的重要组成部分，它有两个用途。

短期来看，Scratchpad 在当前思考中就可以被引用。智能体可以查看草稿纸，确认已经尝试了哪些方案，避免重复。

长期来看，Scratchpad 是生成 Working Notes 的重要素材。当触发压缩时，系统会读取所有草稿纸条目，把它们作为输入传递给大语言模型。

Scratchpad 的内容被标记为"工作过程中的自留笔记"，它们可能与 Session History 有重复，但它们标记了智能体认为重要的信息。

压缩完成后，系统会清空 Scratchpad。因为草稿纸的内容已经被整合到 Working Notes 中，继续保留会重复。清空 Scratchpad 为下一阶段任务做准备。

## 压缩的执行流程

压缩在执行循环的每次迭代开始时进行检查。

当满足压缩条件时，系统会：
1. 调用 _generate_working_notes() 生成 Working Notes
2. 对于 Top-level MicroAgent，调用 _push_pending_summary() 把待总结 messages 推送到 root_agent 的队列
3. 重新构建 messages 列表
4. 清空 Scratchpad

整个过程是同步的，会暂停正常的思考流程。但由于压缩通常在会话已经比较长时才触发，而且压缩后的 messages 会显著变短，总体上对性能的影响是正面的。

## 压缩后的 messages 结构

压缩后的 messages 列表变得非常简洁。

如果有 system message，它会保留在第一个位置。这是 System Prompt，定义了智能体的角色和能力。

第二个位置是一个 user message，包含了压缩后的内容：
- 对于 Top-level MicroAgent：Email History（所有邮件的紧凑格式）+ Working Notes
- 对于嵌套 MicroAgent：原始任务内容 + Working Notes

这个结构的优势在于：
- System Prompt 保持不变，确保智能体的角色定义稳定
- Email History 或原始内容提供了完整的沟通上下文
- Working Notes 提供了执行过程中的关键信息
- 总长度大大缩短，减少了后续思考的 token 消耗

## Working Notes 的生成质量

Working Notes 的质量直接影响压缩的效果。系统通过几种方式确保质量：

首先是使用专门的提示和解析器。提示引导模型进行场景识别、结构定义、状态提取，确保生成的是结构化的、有针对性的笔记，而不是简单的摘要。

其次是使用 think_with_retry 模式。如果模型返回的格式不符合预期，系统会给出反馈，让模型重新生成。这确保了 Working Notes 始终符合预期格式。

然后是分层引导。对于 Top-level MicroAgent，提示明确说明邮件记录始终可读，Working Notes 应该聚焦于执行层、发现层、调整层，避免重复沟通层的内容。这种分层引导让生成的笔记更加精炼和有用。

最后是 Scratchpad 输入。草稿纸中的内容是智能体主动记录的重要信息，它们作为输入帮助模型理解什么是关键的。

## 压缩对性能的影响

压缩对性能有显著的正面影响。

首先是减少了后续思考的 token 消耗。虽然压缩本身需要调用模型（产生一次成本），但后续的每次思考都会受益于更短的上下文。对于一个长会话，压缩的投入是值得的。

其次是提高了响应速度。更短的上下文意味着模型处理更快，用户能更快收到回复。

然后是延长了会话的有效长度。通过压缩，会话可以无限延长，而不会超出模型的上下文限制。这让长时间的任务、多轮的对话成为可能。

潜在的影响是压缩可能丢失一些细节。但 Working Notes 的设计目标是保留继续工作所需的线索——文件路径、中间产物位置、技术决策原因、未完成的子任务。这些线索确保了压缩后的会话仍然可以有效地继续工作。

## 压缩的透明性

从用户的角度，压缩应该是透明的。用户不需要知道会话何时被压缩，也不需要手动触发压缩。系统自动在适当的时候执行压缩，保持对话的流畅性。

但用户可能会注意到一些迹象。比如，当会话比较长后，智能体的某些回复可能更简洁，因为它在引用 Working Notes 而不是详细的对话历史。又比如，智能体可能会偶尔"忘记"很久以前提到的某个细节。

如果用户发现智能体确实遗忘了重要信息，可以主动提醒。用户可以重复之前提到的信息，或者明确告诉智能体"记得之前我们讨论过某某"。智能体会把这些新信息加入会话历史，继续对话。

## 总结

会话自动压缩是 AgentMatrix 管理长对话的关键机制。通过 token 阈值和 Scratchpad 积累触发压缩，使用液态金属架构生成动态结构的 Working Notes，区分 Top-level 和嵌套 MicroAgent 的不同处理方式，系统让会话可以无限延长而不超出模型能力。

Scratchpad 与压缩的协作，让重要的信息始终被保留，而冗余的细节被过滤。压缩后的 messages 结构简洁而完整，既减少了 token 消耗，又保留了继续工作所需的关键信息。

理解压缩机制，有助于理解为什么智能体在长对话中依然能保持连贯，以及如何有效地使用 Scratchpad 来标记重要信息。
